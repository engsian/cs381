Semantic HTML

Choosing elements by meaning, not appearance

Key Question

"What is this content?" — not "How should it look?"

The Problem: "Div Soup"

Non-Semantic

<div class="header">
  <div class="nav">...</div>
</div>
<div class="main">
  <div class="article">
    <div class="section">...</div>
  </div>
  <div class="sidebar">...</div>
</div>
<div class="footer">...</div>

No meaning to browsers or screen readers

Semantic

<header>
  <nav>...</nav>
</header>
<main>
  <article>
    <section>...</section>
  </article>
  <aside>...</aside>
</main>
<footer>...</footer>

Clear structure, accessible, SEO-friendly

Page Structure Blueprint

<header>
<nav>

<aside>
<main>
<article> <section>

<aside>
<footer>

Each element tells browsers and assistive tech what the content IS

Semantic Elements by Purpose

Structure

  • <header> - Introductory content
  • <nav> - Navigation links
  • <main> - Primary content
  • <footer> - Closing content

Content

  • <article> - Self-contained content
  • <section> - Thematic grouping
  • <aside> - Related but separate

Media

  • <figure> - Illustrations, diagrams
  • <figcaption> - Caption for figure

Text

  • <mark> - Highlighted text
  • <time> - Dates and times
  • <details> - Expandable content
  • <summary> - Details heading

HTML Forms

The gateway to user interaction

User-Centered Design

Good forms reduce friction, guide users, and prevent errors

Anatomy of a Form

<label> — Describes the input (accessibility!)
<input> — Captures user data (many types)
<select> — Dropdown choices
<textarea> — Multi-line text
<button> — Submit or action

Input Types - Choose Wisely

The right type = better UX, mobile keyboards, built-in validation


Text type="text"

Email type="email"

Password type="password"

Number type="number"

Phone type="tel"

Date type="date"

URL type="url"

Search type="search"

Built-in Validation

Let HTML do the work—no JavaScript needed!

<form>
  <!-- Required field -->
  <input type="text" required>

  <!-- Email format -->
  <input type="email" required>

  <!-- Number range -->
  <input type="number" min="1" max="100">

  <!-- Text length -->
  <input type="text" minlength="3" maxlength="50">

  <!-- Pattern matching -->
  <input type="text" pattern="[A-Za-z]{3,}"
         title="At least 3 letters">
</form>
Browser shows error messages automatically!

Form Design Thinking

Ask Yourself

  • What's the minimum data I need?
  • What input type makes sense?
  • Can I provide helpful defaults?
  • What errors might users make?

Best Practices

  • Always use <label>
  • Group related fields with <fieldset>
  • Use placeholder as hint, not label
  • Provide clear error messages

ARIA - Enhancing HTML

Accessible Rich Internet Applications

<!-- Role: what is this element? -->
<div role="navigation">...</div>

<!-- Label: what does it say? -->
<button aria-label="Close menu">
  <i class="fa fa-times"></i>
</button>

<!-- State: what's happening? -->
<button aria-expanded="false">Menu</button>

<!-- Description: more context -->
<input aria-describedby="help-text">
<p id="help-text">Enter your email</p>
Use semantic HTML first—ARIA is for when HTML isn't enough

Quick Accessibility Wins

1 Add alt text to all images
2 Use proper heading hierarchy (h1 → h2 → h3)
3 Always associate <label> with inputs
4 Ensure sufficient color contrast
5 Make all functionality keyboard accessible

Web Storage Example

// Save data (persists after browser close)
localStorage.setItem('username', 'Ahmed');
localStorage.setItem('theme', 'dark');

// Retrieve data
const user = localStorage.getItem('username');
const theme = localStorage.getItem('theme');

// Remove data
localStorage.removeItem('username');

// Clear all
localStorage.clear();

// Session storage (cleared when tab closes)
sessionStorage.setItem('cartItems', '5');
Use cases: Remember preferences, save form data, shopping cart

HTML Templating

Why we need reusable structures

The Problem with Plain HTML

Repetition

Same header/footer on 50 pages? Copy-paste nightmare.

No Variables

Can't say "Hello, {username}" - HTML is static text.

No Logic

Can't show/hide content based on conditions.

No Loops

100 products? Write 100 <div>s manually.

Real-World Scenario

Imagine: Your site has 50 pages, all with the same navigation bar.

The client says: "Add a new menu item."

Plain HTML

Edit 50 files manually
Risk missing one
Time: hours

With Templates

Edit 1 file
Changes everywhere
Time: seconds

Templating Solutions

Server-Side (PHP)

<?php include 'header.php'; ?>

<h1>Welcome, <?= $username ?></h1>

<?php foreach($products as $p): ?>
  <div class="product">
    <h3><?= $p['name'] ?></h3>
    <p>$<?= $p['price'] ?></p>
  </div>
<?php endforeach; ?>

<?php include 'footer.php'; ?>

Client-Side (JS)

// Template literal
const card = (product) => `
  <div class="product">
    <h3>${product.name}</h3>
    <p>$${product.price}</p>
  </div>
`;

// Render all products
products.map(card).join('');

The <template> Element

HTML5's built-in templating feature

<!-- Hidden template - not rendered -->
<template id="card-template">
  <div class="card">
    <h3 class="title"></h3>
    <p class="description"></p>
  </div>
</template>

<script>
// Clone and use the template
const template = document.getElementById('card-template');
const clone = template.content.cloneNode(true);

clone.querySelector('.title').textContent = 'My Card';
clone.querySelector('.description').textContent = 'Details...';

document.body.appendChild(clone);
</script>
